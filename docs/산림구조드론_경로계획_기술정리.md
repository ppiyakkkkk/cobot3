# 산림 인명구조 드론 경로 계획 알고리즘 기술 정리

## 1. 개요

- **목적**: 산림 환경에서의 인명구조 임무를 수행하는 드론의 경로 생성(전역) 및 경로 추종/회피(지역) 알고리즘 정리
- **플랫폼**: 쿼드로터 드론 N대 (전방향 이동 가능, N은 설정값) — 전 지역을 N대가 구역을 나눠 동시 수색
- **시뮬레이션 환경**: NVIDIA Isaac Sim
- **비행 제어**: Pegasus Simulator
- **내비게이션 프레임워크**: ROS2 Nav2
- **환경 특성**:
  - 2.5D 지형: 정확히는 3D 장애물 정보를 드론의 비행 고도 밴드로 필터링해 2D costmap에 투영하는 방식 (임의 고도로 자유롭게 상하 회피하는 완전 3D는 아님)
  - 지면 자체의 고저차(산악 지형)로 인해 드론의 절대 고도는 위치에 따라 변함 → 지면 상대 고도 보정 필요 ([7장](#7-지형-고저차-대응-지면-상대-고도-보정) 참고)
  - 나뭇가지 등 세밀한 장애물이 맵에 포함된 상황을 1차 목표로 함
  - 이후 간략화된 지형지물 맵만 주어지는 상황에서 실시간 장애물 반영으로 확장 예정 (추가 과제)

---

## 2. 요구사항 요약

| 항목 | 요구사항 |
|---|---|
| 경로 생성 방식 | 전역 경로(Global) / 지역 경로(Local) 분리 |
| 프레임워크 제약 | Nav2 표준 플러그인 내에서 선택 |
| 환경 차원 | 2.5D (높이 정보 포함 평면 기반) |
| 장애물 특성 | 나뭇가지 등 세밀한 구조물 회피 필요 |
| 확장 요구사항 | 정밀 맵이 없는 경우, 실시간 센서 기반 장애물 반영 필요 |
| 드론 대수 | N대(설정값) 동시 수색. 드론별 담당 구역은 공간적으로 분리(겹침 없음)하고 총 이동거리가 균형이 되도록 배분 |
| 수색 범위 | 전역 경로(드론별 담당 구역 포함)만 따라가도 전 지역 커버 보장 (로컬 회피로 인한 일부 누락은 추후 과제) |

## 3. 맵 생성 파이프라인 (Map Generation Pipeline)

전역 경로 계획(4장)과 Costmap 구성(6장)에서 사용하는 모든 맵 데이터가 어디서, 어떤 순서로 만들어지는지 정리한다.

### 3.1 파이프라인 개요

```
[전역 USD] → [전역 3D 점유맵] → [전역 3D 코스트맵] → [전역 2D 코스트맵] → [로컬 2D 코스트맵]
                                                              │              ↑
                                                              │    드론 LiDAR(실시간, rolling window)
                                                              ▼
                                                  [전역 커버리지 웨이포인트]
                                                              │
                                                     드론 대수 N만큼 구간 분할
                                                              ▼
                                                 [드론별 웨이포인트 (드론 1..N)]
```

| 단계 | Input | 생성 방법 | 활용처 |
|---|---|---|---|
| 1. 전역 USD | 산림 환경 3D 모델(지형+수목) | Isaac Sim USD 씬 저작 | 맵이 아닌 소스 환경. 이후 모든 맵 생성의 출발점 |
| 2. 전역 3D 점유맵 | 전역 USD의 지형/장애물 메쉬 | 메쉬 복셀화(voxelization) → free/occupied 3D grid | 3D 코스트맵 생성 기반. 지면 레이어만 추출 시 [DEM](#7-지형-고저차-대응-지면-상대-고도-보정)의 소스로도 재사용 |
| 3. 전역 3D 코스트맵 | 전역 3D 점유맵 | 3D 거리변환(distance transform)으로 장애물 근접 비용 부여 | 2D 코스트맵 생성 기반 |
| 4. 전역 2D 코스트맵 | 전역 3D 코스트맵 | DEM 기준 지면 상대 고도 밴드(지면+5m)로 슬라이스/투영 | Nav2 Global Costmap Static Layer로 로드 → Theta\* 전역 경로 계획(4장)에 사용 |
| 5. 로컬 2D 코스트맵 | 전역 2D 코스트맵 + 드론 탑재 LiDAR 실시간 포인트클라우드 | Nav2 Local Costmap: Static Layer(4단계 결과) + Voxel Layer(DEM 기준 지면 상대 고도로 변환된 실시간 데이터, rolling window) | MPPI 지역 경로/장애물 회피(5장)에 사용 |
| 6. 전역 커버리지 웨이포인트 | 전역 2D 코스트맵 | 자유공간 바운딩 박스를 지그재그(보루스트로페돈)로 스윕하는 순서대로의 좌표 리스트 생성. 줄 간격은 카메라 풋프린트 폭 기준, 각 줄은 시작/끝 코너 2점만 사용 (3.3절 상세) | 7단계 입력. 드론이 1대뿐이라면 이 리스트를 그대로 사용 |
| 7. 드론별 웨이포인트 | 전역 커버리지 웨이포인트 + 드론 대수 N + 공통 출발점 | 줄 목록을 N개의 연속 구간으로 분할하고 이동거리 균형을 보정 (3.3절 상세) | 드론별 `nav2_waypoint_follower`가 자기 몫의 리스트를 순회하며 4장 Theta*를 매 구간마다 호출하는 목적지 리스트로 사용 → 드론들이 구역을 나눠 맡아도 전역 경로만 따라가면 전 지역 커버 보장(2장 요구사항) |

### 3.2 기존 아키텍처와의 관계

- 이 장은 맵이 **어떻게 만들어지는지**(생성 측)를 다루고, 6장(Costmap 구성)은 **Nav2가 어떻게 소비하는지**(런타임 레이어 구성)를 다룬다.
- **전역** 코스트맵의 3D→2D 투영은 4단계(오프라인)에서 이미 끝나 있으므로, Nav2에는 완성된 2D 그리드가 Static Layer로 로드된다. 6장의 Voxel Layer는 **로컬** 코스트맵 전용이다.
- 4단계(z-band 슬라이스)에서도 [7장](#7-지형-고저차-대응-지면-상대-고도-보정)에서 설명하는 절대/상대 높이 문제가 동일하게 발생하므로, DEM 기반 상대 높이 보정을 이 단계에도 적용해야 한다.
- 이 파이프라인은 **x-y 평면상의 장애물 회피용 코스트맵 생성만 담당**한다. 드론이 지면 기준 5m 고도를 유지하도록 실제 z(고도)를 움직이는 것은 코스트맵과 무관한 별도 고도 제어 모듈(7.3)이 전담한다.
- 산출물 포맷: 전역 2D 코스트맵은 `nav2_map_server` 표준 포맷(YAML+PGM, `trinary_costmap: false`로 그레이디드 값 반영), DEM은 `.npy`(float32)+`.yaml`(resolution/origin/width/height)로 저장한다 (경로·파일명은 `system_architecture.md` 참고). 2장 요구사항(Nav2 표준 플러그인)에 따라 커스텀 코스트맵 플러그인은 쓰지 않는다.
- 6~7단계(전역 커버리지 웨이포인트 생성, 드론별 배분)는 Nav2 플러그인이 아닌 커스텀 오프라인 로직이다. 2장의 "Nav2 표준 플러그인 내에서 선택" 요구사항은 전역/지역 경로 실행 알고리즘(Theta*/MPPI)에 대한 것이며, 이 전처리 단계에는 적용되지 않는다.

### 3.3 커버리지 웨이포인트 생성 알고리즘 상세 (다중 드론 포함)

#### 3.3.1 스윕 라인 생성 (6단계)

- 자유공간(free space) 바운딩 박스에 대해 보루스트로페돈(지그재그) 스윕 라인을 생성한다.
- **줄 간격(row spacing) = 카메라 풋프린트 폭**: 고도 5m·수평 FOV 60°·아래쪽 30° 틸트로 지면에 투영되는 폭을 기준으로 계산한다. 카메라가 아래쪽으로 기울어져 있어 실제 풋프린트는 사다리꼴(근거리 폭 < 원거리 폭)에 가까우므로, 빈틈이 생기지 않도록 보수적으로 좁은 쪽(근거리) 폭을 줄 간격으로 채택한다. 정확한 사다리꼴 수식은 구현 단계에서 확정한다.
- 각 줄은 **시작/끝 코너 2점만으로 표현**한다. 줄 내부를 촘촘한 점으로 채울 필요가 없는 이유: 카메라는 비행 중 연속 촬영하고, 두 코너 사이는 Theta*가 잇는 직선에 가까운 경로를 따라 자연히 커버되기 때문이다.
- 결과: 줄1 시작→줄1 끝→줄2 끝→줄2 시작→... 순서의 단일 좌표 리스트 (3.1절 6단계 산출물).
- 이 항목은 전제상 자유공간이 사각형에 가까운 단순 볼록 영역(나무 등은 점 장애물로 국소 회피)이라고 가정한다. 실제 숲이 이보다 조밀해 자유공간이 조각(미로형)으로 갈라지면 재검토가 필요하다(3.3.4 참고).

#### 3.3.2 드론 대수 N만큼 구간 분할 (7단계)

- 입력: 3.3.1의 전체 줄 목록(줄 개수 R), 드론 대수 N(설정값), 공통 출발점(모든 드론이 동일 지점에서 이륙).
- **1차 분할**: R개의 줄을 N등분(R÷N)하고, 나머지는 앞쪽 그룹부터 1줄씩 배정하여 N개의 **연속** 구간으로 나눈다.
- **보정**: 각 드론의 총 이동거리(공통 출발점 → 자기 구간 진입점까지 거리 + 구간 내 스윕 거리)를 계산하고, 그룹 간 최대-최소 격차가 임계치 이상이면 인접 그룹 경계를 ±1줄씩 옮기는 로컬 탐색(hill-climbing)으로 격차를 줄인다.
- 결과: 드론마다 숲을 가로지르는 겹치지 않는 연속된 줄 구간(가로 띠) 하나씩 배정되고, 각 드론의 웨이포인트 리스트는 자기 구간 안에서 3.3.1과 동일한 지그재그 순서를 따른다. 구역이 공간적으로 분리되므로 드론 간 충돌 위험이 구조적으로 낮다.

#### 3.3.3 출력 포맷

- 드론별 웨이포인트 파일로 분리 저장한다: `forest_world_waypoints_drone{i}.yaml` (i=0..N-1). 포맷은 기존 커버리지 웨이포인트 YAML과 동일한 좌표 리스트다.

#### 3.3.4 이번 설계의 전제와 남은 과제

- 자유공간이 사각형에 가까운 단순 볼록 영역이라는 전제(2장 참고) 하의 설계다. 실제 숲 에셋(예: `korean_mountain_100x100_dense_grouped.usd`, 활엽수·침엽수·덤불·바위가 모두 충돌체로 포함될 예정)의 밀도를 아직 확인하지 못했다 — 실제로 조밀해 자유공간이 여러 조각으로 갈라지면 "연속 줄 구간 = 겹치지 않는 구역" 전제가 깨져 셀 분할(cellular decomposition) 등으로 재설계해야 할 수 있다.
- 드론 대수 N이 줄 개수 R보다 많은 경우(N > R)는 이 설계에서 다루지 않는다.
- 카메라 풋프린트의 정확한 사다리꼴 수식(근거리/원거리 폭 차이 반영)은 구현 단계에서 확정한다.

## 4. 전역 경로 생성 (Global Planner)

전역 경로의 목적지는 임의의 단일 지점이 아니라, 3장 6~7단계에서 생성되고 드론 대수 N만큼 나뉜 커버리지 웨이포인트 리스트다. 드론마다 자기 몫의 리스트를 가지고, 각자의 `nav2_waypoint_follower`가 이 리스트를 순서대로 순회하며 매 구간(현재 위치 → 다음 웨이포인트)마다 Theta*를 새로 호출한다. 드론들이 겹치지 않는 구역을 나눠 맡아도, 전체 드론의 경로를 합치면 지정된 구역이 빠짐없이 커버되도록 한다.

### 4.1 후보 비교

| 알고리즘 | Nav2 패키지 | 특징 | 산림 환경 적합성 |
|---|---|---|---|
| **Theta\*** | `nav2_theta_star_planner` | Any-angle 경로, 그리드에 구애받지 않는 자연스러운 대각선 이동 | 나무 사이를 매끄럽게 통과, 계단식 경로 없음 |
| Smac Planner (2D) | `nav2_smac_planner` | A* 기반 + 후처리 스무딩 | 세밀한 costmap 해상도에서도 안정적이나 Theta*보다 경로가 덜 자연스러움 |
| Smac Planner (Hybrid-A\*) | `nav2_smac_planner` | 헤딩/운동학적 제약 반영 | 쿼드로터는 전방향 이동 가능하므로 불필요한 제약으로 작용 |

### 4.2 선정: Theta\*

- 쿼드로터는 헤딩 제약이 없는 전방향 이동체이므로 Hybrid-A*의 제약이 불필요
- Any-angle 특성으로 나뭇가지 사이의 좁고 불규칙한 공간에서 실제 장애물 형상에 맞는 경로 산출
- 그리드 기반 A* 대비 경로 길이와 회전 횟수가 줄어들어 비행 효율성 확보

## 5. 지역 경로 / 추종 (Local Planner / Controller)

### 5.1 후보 비교

| 알고리즘 | Nav2 패키지 | 특징 |
|---|---|---|
| **MPPI** | `nav2_mppi_controller` | 샘플링 기반 다수 궤적 시뮬레이션 후 costmap 비용 평가, 실시간 회피에 강함, 비선형 동역학 반영 가능 |
| DWB | `nav2_dwb_controller` | 속도 공간 샘플링 기반, 상대적으로 단순한 환경에 적합 |
| TEB | 별도 패키지 (`teb_local_planner`, Nav2 공식 저장소 외부) | 좁은 공간 회피에 강하지만 추가 설치 필요 |

### 5.2 선정: MPPI Controller

- 매 스텝 다수의 궤적 후보를 시뮬레이션하여 local costmap에 새로 반영된 장애물(나뭇가지 등)에 즉각 반응 가능
- 기본 제공되는 Omni 모션 모델은 2D 평면 속도(vx, vy, wz) 기반이라 드론의 3D 추력/자세 동역학을 그대로 반영하진 않음 → 관성/기동 특성을 살리려면 Omni 모델 기반에 커스텀 critic 및 파라미터 튜닝 추가 필요
- Nav2 공식 저장소 내 포함되어 추가 패키지 설치 없이 사용 가능

## 6. Costmap 구성 (장애물 반영 핵심 요소)

알고리즘 선택 이상으로, 세밀한 장애물(나뭇가지)을 반영하려면 costmap 레이어 설계가 핵심.

| 레이어 | 역할 | 적용 대상 |
|---|---|---|
| **Static Layer** | 3장 파이프라인에서 이미 3D→2D로 투영/완성된 전역 2D 코스트맵을 그대로 로드 | Global Costmap + Local Costmap (로컬도 이 맵을 기본 바탕으로 깔고 시작) |
| **Voxel Layer** | 드론 탑재 LiDAR의 실시간 3D 포인트를 비행 고도 범위로 걸러 2D costmap에 투영 (rolling window). 나뭇가지처럼 특정 높이에만 존재하는 장애물을 지면과 구분하여 반영 | Local Costmap (Static Layer 위에 실시간 데이터를 덮어씀) |
| **Inflation Layer** | 얇은 장애물(가지 끝 등) 주변에 안전 마진 부여. 마진이 없으면 경로가 장애물에 너무 근접하게 계획될 위험 | Local Costmap (전역은 3장 3D 코스트맵 단계에서 거리 기반 비용이 이미 반영됨) |
| Obstacle Layer | 단순 2D 투영. 수직 구조 구분이 필요 없는 경우 대체 가능하나, 산림 환경에서는 Voxel Layer가 더 적합 | (미사용) |

> **참고 — Costmap이 필요한 이유**: Theta*(전역)와 MPPI(지역) 모두 "장애물이 어디 있는지"를 costmap을 통해서만 참조하는 구조. 전역 코스트맵은 3장 파이프라인에서 미리 만들어져 Static Layer로 로드되고, 로컬 코스트맵도 같은 Static Layer를 기본으로 깔되 그 위에 Voxel/Inflation Layer가 실시간으로 얹힌다(센서 사각지대·범위 밖 구역도 정적 맵으로 대략 인지 가능). costmap은 위치 추정(localization, GPS/IMU/Odometry 기반)과는 무관한 별개 개념이며, "주변에 무엇이 있는지"만 표현함.
>
> **실시간 센서 레이어(Voxel Layer)가 정적 맵만으로도 충분한지**: 나뭇가지가 정적이고 맵이 완전히 정확하다는 전제라면 이론상 Static Layer만으로도 가능하나, (1) 위치 추정 오차 누적, (2) 맵-실제환경 간 미세한 불일치 가능성, (3) 2차 과제(간략 맵)와의 아키텍처 일관성을 고려해 실시간 레이어를 함께 두는 것을 권장. 다만 1차 과제 초기 검증 단계에서는 실시간 레이어 비중을 낮추거나 꺼두고 시작하는 것도 합리적인 단순화.
>
> **전역/지역 비용 스케일 일치 필요**: 전역 코스트맵(Static Layer)의 비용은 3장 3단계 3D 거리변환으로 오프라인에서 미리 계산되고, 지역 코스트맵의 비용은 위 Inflation Layer(Nav2 표준 플러그인)가 실시간으로 계산한다 — 두 계산 방식이 다르면 Theta*가 잡은 전역 경로의 안전 마진과 MPPI가 실시간으로 판단하는 안전 마진이 서로 어긋나, 로컬에서 불필요하게 흔들리거나 반대로 위험을 과소평가하는 경로가 나올 수 있다. 3장 3단계의 거리변환 비용 계산에는 Nav2 InflationLayer와 동일한 공식(`cost = 253 × exp(-cost_scaling_factor × (거리 - inscribed_radius))`)과 동일한 파라미터(`cost_scaling_factor`, `inscribed_radius`)를 적용해, 전역/지역 코스트맵이 같은 기준으로 "위험하다"를 판단하도록 맞춰야 한다.

## 7. 지형 고저차 대응 (지면 상대 고도 보정)

산림 환경은 지면 자체에 고저차가 있고, 드론은 항상 "지면 기준 고도 5m"를 유지하며 비행하는 지형 추종(terrain-following) 방식을 쓴다. 이 때문에 드론의 **절대 z값은 위치에 따라 계속 변한다**. 이 상황에서 표준 설정을 그대로 쓰면 문제가 발생한다.

### 7.1 문제: 3D→2D 높이 필터링은 절대 좌표 기준

- 3D 장애물 데이터를 2D로 투영할 때(전역: 3장 4단계 오프라인 슬라이스 / 지역: `nav2_costmap_2d`의 Voxel Layer) `min_z` / `max_z`를 **월드 좌표계의 절대 높이**로 고정하면 문제가 생긴다
- 지면 고저차가 있는 지형에서 이 값을 고정하면:
  - 언덕 위: 지정한 절대 높이 구간이 비어 있어 장애물 필터링이 무의미해짐
  - 저지대: 반대로 지면 근처를 걸러내는 문제가 생김
- 즉 "지형이 평평하다"는 전제가 깨지는 순간 절대 높이 밴드 필터링 자체가 성립하지 않음

### 7.2 해결 방안 1: 지면 상대 높이로 변환

1. **DEM(지형고도모델) 확보**: 3장 2단계(전역 3D 점유맵)의 지면 레이어에서 추출한 2D 고도 지도 (x, y → 지면 z값)
2. **장애물 데이터 z값을 상대 높이로 변환**: `relative_z = absolute_z - terrain_height(x, y)`
3. 변환된 상대 높이 데이터의 `min_z / max_z`를 지면+5m 중심의 상대 기준(예: 4~6m 밴드)으로 고정
   → 언덕 위/저지대 관계없이 "지면에서 5m 근처"가 항상 올바르게 필터링됨

이 변환의 적용 위치는 전역/지역에서 다르나, **기준은 두 경우 모두 DEM**이다.

| 구분 | 적용 위치 | 방식 |
|---|---|---|
| 전역 (Static Layer) | 3장 4단계(z-band 슬라이스) 내부 | 오프라인 파이프라인에서 DEM 기준 상대 높이로 변환 후 슬라이스 |
| 지역 (Voxel Layer) | Voxel Layer 앞단 | 실시간 LiDAR 포인트클라우드의 **포인트별 (x, y) 위치마다** DEM 지면고도를 조회하여 상대 높이로 변환하는 **포인트클라우드 전처리 노드(ROS2 커스텀 노드)** 추가 (드론 자신의 위치 1곳만 기준으로 삼으면 경사 지형에서 드론과 먼 포인트일수록 오차가 커짐 → DEM을 격자 lookup grid로 미리 준비해 포인트별 조회 비용을 낮춤) |

두 경우 모두 핵심 알고리즘(Theta*, MPPI)은 변경 없이, 데이터 전처리 단계만 추가된다.

### 7.3 해결 방안 2: 수평/수직 역할 분리

- **Nav2 (Theta* + MPPI)**: 7.2에서 상대 높이로 보정된 costmap을 기준으로 x-y 평면 경로 계획 및 회피만 담당
- **별도 고도 제어 모듈 (Nav2 외부)**: DEM을 참조하여 "현재 x,y 위치에서 지면+5m가 절대 몇 m인지"를 실시간 계산 → 목표 고도와 현재 고도의 차이를 z velocity로 변환해 Pegasus/PX4에 전달 (`system_architecture.md`의 `altitude_controller_node` 참고)

이 구조로 Nav2 표준 아키텍처를 그대로 유지하면서, 산악 지형에서의 상하 이동은 별도 로직이 전담하도록 역할을 분리할 수 있다.

### 7.4 요약

| 항목 | 내용 |
|---|---|
| 문제 | 3D→2D 높이 필터링(전역 오프라인 슬라이스 / 로컬 Voxel Layer)이 절대 z 기준이라 지형 고저차 있는 환경에서 오작동 |
| 필요 데이터 | DEM(지면 고도 지도, 3장 2단계 지면 레이어에서 추출) |
| 필요 작업 | 장애물 z값 → 지면 상대 높이 변환 (전역: 오프라인 파이프라인 내 처리 / 지역: 전처리 노드) |
| 아키텍처 원칙 | Nav2는 x,y 회피 전담, z(고도)는 별도 지형추종 모듈이 전담 |

## 8. Isaac Sim / Pegasus 연동 아키텍처

### 8.1 전체 파이프라인 개요

> 전역 2D 코스트맵은 3장 맵 생성 파이프라인에서 오프라인으로 사전 생성되어, 아래 런타임 파이프라인에는 Static Layer로 로드된다. 아래는 드론 1대 기준이며, 실제로는 드론 대수 N만큼 독립적으로(네임스페이스 분리) 반복 실행된다 — 노드/토픽 구성 상세는 `system_architecture.md` 참고.

```
[Isaac Sim]
  - 산림 환경 3D 모델 (지형 + 나뭇가지 등 세밀 장애물)
  - 드론 물리 시뮬레이션
  - 가상 센서 (Depth Camera / LiDAR)
        │
        ▼
[Pegasus Simulator]
  - 드론 동역학 모델링 및 저수준 비행 제어 (PX4 SITL 연동)
  - Isaac Sim ↔ ROS2 브릿지 역할
        │
        ▼
[ROS2 Topics]
  - Odometry (/pegasus/odom 등)
  - PointCloud2 / LaserScan (센서 데이터)
  - TF (map → odom → base_link)
        │
        ▼
[Nav2 Stack]
  - Global Costmap ← Static Layer (3장 파이프라인에서 사전 생성된 전역 2D 코스트맵)
  - Local Costmap ← Static Layer (전역 2D 코스트맵, 기본 바탕) + Voxel Layer (실시간 센서 데이터, rolling window)
  - Planner Server (Theta*) → 전역 경로 생성
  - Controller Server (MPPI) → 지역 회피 및 경로 추종
  - BT Navigator → 전체 내비게이션 흐름 제어
        │
        ▼
[Pegasus] → 속도/자세 명령(cmd_vel 또는 setpoint) 전달
        │
        ▼
[Isaac Sim] 드론 이동 반영
```

### 8.2 연동 시 주요 확인 포인트

| 항목 | 확인 사항 |
|---|---|
| 좌표 프레임 | Isaac Sim/Pegasus 좌표계(ENU/NED 등)와 Nav2 표준 프레임(map, odom, base_link) 간 변환 일치 여부 |
| Odometry 토픽 | Pegasus에서 발행하는 odometry를 Nav2가 기대하는 `nav_msgs/Odometry` 형식/토픽명으로 매핑 |
| 센서 데이터 | Isaac Sim 가상 Depth Camera/LiDAR → PointCloud2 발행 → Voxel Layer 입력 연결 (로컬 코스트맵용) |
| 제어 명령 전달 | Nav2 Controller Server의 출력(`cmd_vel`)을 Pegasus가 이해하는 제어 인터페이스(예: PX4 offboard setpoint)로 변환 |
| 고도 제어 | 2.5D 평면 내비게이션이므로, 별도의 고도 유지 로직(예: 고정 비행고도 또는 지형 추종 고도)이 Nav2 외부에서 필요할 수 있음. 산악 지형에서의 상대 고도 유지 방식은 [7장](#7-지형-고저차-대응-지면-상대-고도-보정) 참고 |

## 9. 정밀 맵 → 간략 맵 확장 시나리오 (추가 과제 대응)

| 구분 | 1차 (정밀 맵 제공) | 2차 (간략 맵 + 실시간 반영) |
|---|---|---|
| 전역 맵 | 나뭇가지 등 세밀 장애물 포함 | 지형지물 수준의 간략 정보만 포함 |
| 전역 경로(Theta*) | 정밀 정보 기반 최적 경로 산출 | 대략적 골격 경로 산출 (문제 없이 동작) |
| 세밀 장애물 대응 | Static Layer로 충분 | Local Costmap(Static Layer는 그대로, Voxel Layer) + 실시간 센서 업데이트에 더 크게 의존 |
| 조정 필요 사항 | - | Local costmap rolling window 크기 확대, 업데이트 주기 단축 검토 |

- 알고리즘 자체는 변경 없이 재사용 가능
- 정적 맵의 정밀도와 무관하게 "local costmap이 실제 장애물 인지 담당, 전역 맵은 참고용 골격" 구조는 동일하게 유지됨
- 1차 과제의 전역 Static Layer는 3장 파이프라인의 산출물이며, 2차 과제에서도 입력 USD의 정밀도만 낮춰 동일 파이프라인을 재사용할 수 있음

- [ ] 포인트클라우드 상대 높이 변환 전처리 노드 구현 담당 및 일정 (로컬 코스트맵용)

- [ ] 전역 맵 파이프라인(3장) 재실행 트리거 기준: USD(산림 환경) 변경 시뿐 아니라, 안전마진(inflation) 파라미터 변경 시에도 3장 3~4·6~7단계(3D 코스트맵 → 2D 코스트맵 → 커버리지 웨이포인트 → 드론별 배분)까지 재실행하도록 트리거에 포함 — 6~7단계는 4단계의 빈 공간(free space)에 의존하므로 4단계가 바뀌면 항상 같이 재생성해야 함 (USD를 자주 교체할 계획이므로 전역 2D 코스트맵까지 매번 재생성됨을 전제)
- [ ] 높이 밴드(지면+5m 밴드)가 나뭇가지로 완전히 막혀 x-y 회피만으로 통과 불가능한 경우의 처리 방안 (수직 회피 미지원에 따른 Theta* 경로 탐색 실패 / MPPI 정체 대응) — 1차 과제 범위 밖의 미해결 문제로 남겨둠
- [ ] 로컬 회피(MPPI)로 인해 스킵된 커버리지 구역의 재탐색 여부 (드론별로 독립 발생 가능) — 현재 1차 과제 범위 밖, 추후 과제
